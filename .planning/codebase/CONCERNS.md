## Security

### [HIGH] No authentication on any API endpoint
Every route (`/api/assets`, `/api/health`, `/api/alerts`, `/api/predictions`, `/api/ml/predictions`, `/api/analytics/fleet`, `/api/copilot/chat`) is publicly accessible with zero auth checks. Any caller on the network can read all fleet data or send arbitrary copilot prompts.

### [MEDIUM] CORS configured with `allow_methods=["*"]`
Overly permissive CORS allows any HTTP method from any origin. Should be restricted to the specific methods each route actually uses.

### [MEDIUM] Hardcoded DB credentials in source code
Database connection strings or credentials appear directly in source files rather than being loaded exclusively from environment variables.

### [MEDIUM] No input validation or rate limiting on `/api/copilot/chat`
The copilot endpoint accepts arbitrary user-supplied text and forwards it to an LLM with no length limits, no content sanitization, and no rate limiting — open to prompt injection and abuse.

## Architecture / Scalability

### [HIGH] All state is in-process memory — no real persistence
The storage layer declares PostgreSQL but never connects to it. All asset state is held in Python dicts in memory. The application cannot be restarted without losing state and cannot scale horizontally.

### [HIGH] Cannot paginate any list endpoint
`/api/assets` and all other list routes return the entire dataset with no pagination, limit, or cursor support. This will break at scale.

### [MEDIUM] All routes are synchronous
FastAPI routes use blocking I/O patterns. Under any real load this will exhaust the worker pool.

---

## ML Integrity

### [HIGH] Circular dependency: XGBoost trained on rule-engine labels
The ML model is trained on data whose labels are generated by the rule engine. The model learns to replicate rules, not detect real failures from ground-truth data. Predictions from the ML layer add no independent signal over the rule layer.

### [HIGH] Hardcoded asset IDs `HT-07` / `HT-11` in ML inference and training
Specific asset identifiers are hardcoded in ML code paths. The model has asset-specific special-casing baked in, which will silently break when the fleet changes.

## Tech Debt

### [HIGH] Synthetic history algorithm duplicated in 3 files with different growth factors
The logic that generates historical asset data exists in at least 3 separate files, each with slightly different numeric constants. Any change to the algorithm must be made in all three places.

### [HIGH] Zero tests — `tests/` is an empty `__init__.py`
There is no test suite. No unit tests, no integration tests, no e2e tests. All critical paths (health scoring, prediction, ML inference, copilot response) are completely untested.

### [MEDIUM] `datetime.utcnow()` used throughout (deprecated in Python 3.12+)
Should be replaced with `datetime.now(timezone.utc)`.

### [MEDIUM] No Python lock file
`requirements.txt` (or equivalent) pins versions but there is no lock file (`poetry.lock`, `pip-compile` output) to guarantee reproducible installs.

### [LOW] Hardcoded dates and labels in UI components
Some frontend components contain hardcoded date strings or label values that should come from data or config.

### [LOW] Non-haul-truck assets always return dummy healthy scores
Assets that are not haul trucks fall through to a hardcoded "healthy" response regardless of their actual state.

---

## Observability

### [HIGH] Zero logging anywhere in the backend
No structured logging, no request logging, no error logging. When something goes wrong in production there is no trail.

### [HIGH] Health endpoint is unconditional `{"status": "ok"}`
`/api/health` always returns healthy regardless of whether the database, ML model, or any dependency is actually reachable.

### [MEDIUM] No metrics or tracing
No Prometheus metrics, no OpenTelemetry tracing, no performance instrumentation of any kind.

---

## Performance

### [MEDIUM] XGBoost model loaded from disk on every prediction request
The model file is read and deserialized on each call to `/api/ml/predictions` with no in-memory caching. Under any meaningful request volume this will be a serious bottleneck.

### [MEDIUM] Analytics recomputes 360 health evaluations on every fleet analytics request
`/api/analytics/fleet` iterates over all assets × 360 time points and recalculates health scores synchronously on every request with no caching.
